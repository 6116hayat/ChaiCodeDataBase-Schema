# 🛗 Smart Elevator Control System — ER Diagram

This README describes the database design for a **multi-building elevator control platform** used in large infrastructures like corporate towers, malls, hospitals, airports, and high-rise residential complexes. It models how elevators handle floor requests, get assigned rides, track movement, and maintain operational health.

---

## 📌 Design Overview

The database handles:

- Multi-building infrastructure management
- Multiple elevators operating within each building
- Floor-level request generation and tracking
- Smart elevator assignment to requests
- Ride lifecycle tracking (request → assignment → trip)
- Elevator status monitoring (idle, moving, maintenance, offline)
- Maintenance tracking and history
- Ride logs for analytics and performance insights

---

## 🔄 Diagram Flow

The ER diagram follows the real-world lifecycle of a request:

### **Floor Request → Assignment → Elevator Movement → Trip Completion**

- **Floor Request** → Generated by users (up/down)
- **Ride Assignment** → System assigns best elevator
- **Elevator** → Executes the movement
- **Ride Trip** → Logs actual journey (start → end)
- **Status Logs** → Track real-time state changes
- **Maintenance** → Tracks downtime and servicing

---

## 🧱 Core Entities

- buildings
- floors
- elevators_shafts
- elevator
- elevator_floor_mapping
- floor_request
- ride_assignments
- ride_trips
- elevator_status
- maintenance_records

---

## 🧠 Key Idea

- `buildings` = physical structures onboarded
- `floors` = vertical levels inside buildings
- `elevators_shafts` = physical shafts holding elevators
- `elevator` = operational unit handling movement
- `elevator_floor_mapping` = defines which floors an elevator serves
- `floor_request` = user-generated requests from floors
- `ride_assignments` = maps request → elevator
- `ride_trips` = actual execution of rides
- `elevator_status` = tracks elevator state over time
- `maintenance_records` = logs servicing and downtime

---

## 🔗 Relationships Summary

- One building contains multiple floors, shafts, and elevators

- One shaft contains one elevator

- One elevator can serve multiple floors

- One floor can be served by multiple elevators

- One floor request is assigned to one elevator

- One elevator completes multiple ride trips

- One elevator has multiple status logs

- One elevator has multiple maintenance records

## ⚠️ Notes / Improvements

Elevator status is tracked using time-based logs instead of static fields

Ride data is separated from elevator entity to avoid overload

Floor-to-elevator mapping enables support for zoned or express elevators

Maintenance records preserve full history without overwriting

System can be extended with AI-based elevator dispatch logic

Support for zones (low-rise/high-rise) can be added

Real-time monitoring dashboards can be built using status logs
